# Website audit — v2.2 migration findings

*April 2026 · pre-launch readiness audit · sister doc to `06_marketing/master-plan/phase0-marketing-plan.md` v2.2*

---

## 1. Executive summary

**The website is structurally complete but commercially stale.** Every public-facing surface — six LP variants, 18 canonical pages, all 10 Customer.io email sequences, multiple React components, the SEO metadata, and the OpenGraph image — references the legacy v1 pricing (£29 / £44 / £69) and the retired commission/discount model (20% off subscriber retests, 15% off retest from supplement page).

**Zero mentions of the v2.2 pricing (£99 / £119 / £179)** appear anywhere in `09_website-app/frontend/`.

This is not a small fix. The migration requires synchronised edits across HTML, React (`.tsx`), markdown copy decks, email sequence files, and SEO/schema metadata. If the site goes live before this is done, the affiliate programme is broken: PTs and influencers tell their audiences to use a 10% code worth £11.90 off Kit 2, but the kit is priced at £44 on the website — at which point the discount produces a £4.40 saving instead of £11.90, and "the kit is £119" line in every Story script and Reel template is contradicted on landing.

This is also part of why the affiliate programme cannot meaningfully launch DMs at scale until the site migration is shipped. The two are coupled: the asset pack assumes v2.2 pricing on landing.

## 4. Risk flags beyond pricing

Three things found incidentally that aren't pricing but warrant flagging:

**R1 — Compliance language on supplement pages.** The "15% off your next blood test kit" framing on Daily Stack and Collagen pages is a Section D violation under v2.2 — discount > 10% by promotional copy. Even if you fix it to "10% off," the supplement-page mechanic of routing customers to a discount on a kit is a compounding-discount risk: a customer using a PT's 10% code who also has a "10% subscriber retest" benefit might think they get 20%. Ambiguity here is an ASA exposure. Recommend: kill the supplement-page discount offer entirely until v3 — the PT/affiliate code is the only customer-facing discount mechanic in v2.2.

**R2 — Kit 1 panel definition diverges.** Email seq-03a references markers (`Free Androgen Index`, `Albumin`) that aren't in the v2.2 marketing context glossary (`Total T, SHBG, Free T`). One of these is wrong. The product team needs to lock the canonical Kit 1 panel and propagate. Until resolved, do not publish either email or marketing context.

**R3 — `lp-architecture.md` says LP-variant CTAs use legacy prices** ("Order the Testosterone Health Check → £29"). This is the architecture document — anyone implementing should reference it. Update it concurrently with the LPs themselves.

## 5. What this audit didn't cover

- **Performance, accessibility, SEO Core Web Vitals.** Worth running Lighthouse before launch but not blocking.
- **Schema markup on kit pages.** `seo-schema-plan.md` exists and was not audited line-by-line. Verify Product schema reflects v2.2 prices before submitting to Google Search Console.
- **Stripe product setup.** Verify each kit has a Stripe Price object at v2.2 price and that the checkout uses those, not the legacy ones. Likely lives in the backend, not the frontend audited here.
- **Affiliate-code URL handling.** Confirm `?promo=PTNAME15` (or whatever URL param) actually applies the FirstPromoter coupon at checkout. This is the load-bearing piece for the entire affiliate model — one untested checkout flow blocks every Story-coded sale.
- **Page templates and design system.** Visual-only pages (terms, privacy, FAQ) were not checked for pricing references. Recommend a final regex sweep before launch.

---

## 6. The bottom line

The site is well-built. The structure (canonical + LP variants split, six LP variants pre-built, ten email sequences scaffolded) is exactly what the master plan calls for. None of that is in question.

What's in question is the **commercial layer**: prices, discounts, and the lab-turnaround SLA. Those are stale or contradicted across the entire surface. The migration is mechanical — it's editing — but it has to be done synchronously because partial migration is worse than not starting (a £99 hero with a £29 sticky CTA confuses every visitor and tanks the conversion rate).

The migration should run in parallel with — not before — the affiliate-recruitment work I've shipped this week. If you start cold DMs in Week -8 (now), the website doesn't need to be migrated for those DMs to land usefully. By Week -3 to -2, when prospects who replied are being moved to onboarding and codes are about to go live, the migration must be done.

If pricing isn't 100% locked yet — Keith mentioned this — then consider holding the migration until pricing is confirmed by Vitall/lab-partner cost-model, then doing the full migration in one synchronised sweep rather than migrating twice.
